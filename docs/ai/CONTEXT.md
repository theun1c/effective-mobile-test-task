# CONTEXT.md

## Описание проекта

Проект — REST-сервис для агрегации данных об онлайн-подписках пользователей.

Сервис хранит записи о подписках и позволяет считать суммарную стоимость подписок за выбранный период.

---

## Цель проекта

Реализовать backend-сервис на Go для работы с подписками пользователей.

Основные задачи:

- предоставить API для CRUDL-операций над подписками;
- предоставить API для подсчёта суммарной стоимости подписок;
- хранить данные в PostgreSQL;
- запускать сервис через Docker Compose;
- описать API через Swagger.

---

## Основная сущность

### Подписка

Подписка — это запись о месячной онлайн-подписке пользователя.

Поля подписки:

- `service_name` — название сервиса, предоставляющего подписку;
- `price` — стоимость месячной подписки в рублях;
- `user_id` — ID пользователя в формате UUID;
- `start_date` — дата начала подписки в формате месяц-год;
- `end_date` — опциональная дата окончания подписки.

Пример подписки:

```json
{
  "service_name": "Yandex Plus",
  "price": 400,
  "user_id": "60601fee-2bf1-4721-ae6f-7636e79a0cba",
  "start_date": "07-2025"
}
```

---

## Основные фичи

### feat-001-subs

Фича для управления подписками.

Должна поддерживать:

- создание подписки;
- получение подписки;
- получение списка подписок;
- обновление подписки;
- удаление подписки.

---

### feat-002-sum-cost

Фича для подсчёта суммарной стоимости подписок.

Должна поддерживать:

- выбор периода;
- фильтрацию по `user_id`;
- фильтрацию по `service_name`;
- возврат итоговой суммы в рублях.

---

## Технические требования

- Язык: Go.
- API: REST.
- База данных: PostgreSQL.
- Миграции: обязательны.
- Конфигурация: через `.env` или `.yaml`.
- Документация API: Swagger.
- Запуск: Docker Compose.
- Логирование: обязательное.

---

## Важные правила

- Проверка существования пользователя не требуется.
- Управление пользователями не входит в зону ответственности сервиса.
- Стоимость подписки хранится целым числом рублей.
- Копейки не учитываются.
- Даты подписок используются в формате `MM-YYYY`.

---

## Что не входит в проект

Не нужно реализовывать:

- регистрацию пользователей;
- авторизацию;
- роли пользователей;
- платежи;
- интеграции с реальными сервисами;
- проверку существования пользователя;
- уведомления;
- личный кабинет пользователя.

---

## Ожидаемый результат

В результате должен получиться небольшой REST-сервис, который можно:

- запустить через Docker Compose;
- проверить через Swagger;
- использовать для создания и просмотра подписок;
- использовать для подсчёта суммы подписок за период.
