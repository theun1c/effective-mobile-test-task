# FLOW.md

## Назначение

Пошаговый план реализации `feat-002-sum-cost` небольшими инкрементами.

Цель файла:

- вести разработку по порядку;
- не смешивать `sum-cost` с доработками `feat-001-subs`;
- после каждого шага проверять результат;
- использовать файл как опорный сценарий для AI-агента.

---

## Границы фичи

Реализуем только подсчёт суммарной стоимости подписок за период.

Что входит:

- HTTP-ручка `GET /subscriptions/total`;
- параметры `from` и `to`;
- опциональные фильтры `user_id` и `service_name`;
- расчёт суммы по месяцам;
- учёт частичного пересечения периода;
- корректная обработка `end_date = null`;
- логи;
- Swagger для новой ручки.

Что не входит:

- CRUD подписок;
- новые таблицы и новые сущности;
- авторизация;
- пользователи;
- платежи;
- внешние интеграции;
- изменение контракта уже существующих CRUDL-ручек.

---

## Архитектурный подход

Используем тот же поток, что и в `feat-001-subs`:

```text

  ↓

  ↓

  ↓

```

Фича должна переиспользовать уже существующие решения, если они есть:

- тип YearMonth;
- общий формат ошибок;
- общий router;
- общий logger;
- существующую таблицу subscriptions.

———

## Ключевые бизнес-правила

- Подписка участвует в расчёте, если активна хотя бы один месяц внутри периода запроса.
- Стоимость считается по месяцам.
- Количество месяцев считается включительно.
- Если end_date отсутствует, подписка считается активной до бесконечности вправо.
- Если подписка активна только на части периода, учитывается только пересечение.
- Итоговая сумма возвращается целым числом рублей.

———

## Решения, которые нужно держать консистентными

- from и to принимаются строго в формате MM-YYYY.
- Внутри приложения период лучше переводить в нормализованные даты первого дня месяца.
- Алгоритм пересечения должен быть единым для всех сценариев.
- user_id, если передан, валидируется как UUID.
- service_name, если передан, не должен быть пустой строкой.
- Результат API должен быть минимальным:

{
"total_cost": 2400
}

- Ошибки должны возвращаться в том же формате, что и остальной API.

## Шаг 3. Алгоритм расчёта пересечения периода

### Цель

Зафиксировать и протестировать логику подсчёта активных месяцев внутри выбранного периода.

### Файлы

Создать:

- internal/service/subscription_total/period.go
- internal/service/subscription_total/period_test.go

### Что должно появиться

Функции или небольшой модуль для:

- определения пересечения периода подписки и периода запроса;
- расчёта количества активных месяцев;
- обработки end_date = nil;
- расчёта вклада одной подписки в итоговую сумму.

### Обязательные сценарии

- подписка полностью внутри периода;
- подписка начинается раньше периода и заканчивается внутри;
- подписка начинается внутри периода и заканчивается позже;
- подписка покрывает весь период;
- подписка без end_date;
- подписка вообще не пересекается с периодом;
- подписка активна ровно один месяц.

### Проверка после шага

- [ ] количество месяцев считается включительно;
- [ ] непересекающиеся периоды дают 0;
- [ ] end_date = nil обрабатывается корректно;
- [ ] unit-тесты покрывают крайние случаи.

### Риск

Это самый важный кусок фичи. Если тут ошибиться на 1 месяц, дальше ошибка попадёт и в service, и в
SQL, и в API.

## Шаг 4. Repository для суммарного расчёта

### Цель

Добавить в repository метод получения суммы по периоду и фильтрам.

### Файлы

Создать или дополнить:

- internal/repository/postgres/subscription_total_repository.go
- internal/repository/postgres/subscription_total_repository_test.go

### Что должно появиться

Метод repository для подсчёта суммы с учётом:

- периода from / to;
- user_id optional;
- service_name optional;
- активных подписок без end_date.

### Важное решение

Нужно выбрать один подход и использовать его последовательно:

#### Вариант A. Считать в SQL

Плюсы:

- меньше данных гоняется из БД;
- ближе к реальному агрегатному запросу.

Минусы:

- сложнее проверить и поддерживать;
- выше риск ошибки в расчёте месяцев.

#### Вариант B. Вычитывать подходящие подписки и считать в Go

Плюсы:

- проще проверить и отлаживать;
- алгоритм расчёта прозрачнее.

Минусы:

- менее эффективно на больших объёмах.

### Рекомендация

Для тестового задания лучше приоритетно выбрать самый простой и проверяемый вариант. Если данных
немного, допустим расчёт в Go после выборки релевантных подписок. Если уже выбран SQL-first подход в
проекте, тогда считать в SQL, но отдельно тщательно тестировать формулу.

### Проверка после шага

- [ ] repository учитывает период;
- [ ] repository умеет фильтровать по user_id;
- [ ] repository умеет фильтровать по service_name;
- [ ] repository корректно работает с end_date = null;
- [ ] тесты покрывают базовые сценарии.

### Не делать на этом шаге

- HTTP;
- Swagger;
- лишние изменения в CRUDL repository.

## Основные риски

- Ошибка на один месяц из-за некорректной логики включительности периода.
- Разное понимание того, как считать подписку, которая пересекает период только частично.
- Дублирование логики расчёта между Go и SQL.
- Расхождение между тем, что считает repository, и тем, что ожидает service.
- Неявная трактовка end_date = null.
- Swagger может задокументировать одно поведение, а код реализовать другое.
