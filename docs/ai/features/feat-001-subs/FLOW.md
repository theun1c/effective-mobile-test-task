# FLOW.md

## Назначение

Пошаговый план реализации `feat-001-subs` с небольшими инкрементами.

Цель файла:

- вести разработку по порядку;
- не выходить за рамки фичи;
- после каждого шага проверять результат;
- использовать файл как опорный сценарий для AI-агента.

---

## Границы фичи

Реализуем только управление подписками:

- создание подписки;
- получение подписки по ID;
- получение списка подписок;
- обновление подписки;
- удаление подписки.

Не реализуем в рамках этой фичи:

- авторизацию;
- пользователей;
- платежи;
- внешние интеграции;
- подсчёт суммарной стоимости подписок.

---

## Архитектурный подход

Используем простой поток:

```text
HTTP Handler
  ↓
Service
  ↓
Repository
  ↓
PostgreSQL
```

Дополнительно нужны:

- DTO для API;
- валидация входных данных;
- тип/утилита для работы с датой MM-YYYY;
- миграции;
- Swagger;
- Docker Compose;
- логи.

———

## Решения, которые нужно держать консистентными

- price хранится как целое число рублей.
- user_id и id подписки — UUID.
- start_date и end_date приходят/возвращаются в формате MM-YYYY.
- Внутри приложения и в БД даты лучше нормализовать в дату первого дня месяца.
- PUT /subscriptions/{id} трактуем как полное обновление записи.
- GET /subscriptions пока без пагинации.
- GET /subscriptions должен возвращать детерминированный порядок, например created_at DESC.

## Шаг 4. DTO и валидация

### Цель

Отделить API-контракты от доменной модели и централизовать проверки входных данных.

### Файлы

Создать:

- internal/dto/subscription.go
- internal/validation/subscription.go
- internal/validation/subscription_test.go

### Что должно появиться

DTO:

- CreateSubscriptionRequest

Валидация:

- service_name обязателен и не пустой;

- user_id валидный UUID;

- end_date >= start_date.

### Проверка после шага

- [ ] валидный payload проходит;
- [ ] пустой service_name отклоняется;
- [ ] price <= 0 отклоняется;
- [ ] невалидный UUID отклоняется;
- [ ] невалидные даты отклоняются;
- [ ] end_date < start_date отклоняется.

### Важно

Handler не должен сам содержать полную бизнес-валидацию.

## Основные риски

- хранение дат строкой вместо нормализованной даты усложнит следующую фичу;
- отсутствие единого формата ошибок API приведёт к разнобою;
- неясная трактовка PUT как partial/full update приведёт к несовместимому поведению;
- отсутствие фиксированной сортировки в GET /subscriptions сделает ручные проверки нестабильными;
- если сразу не вынести валидацию и тип даты отдельно, логика быстро начнёт дублироваться.
