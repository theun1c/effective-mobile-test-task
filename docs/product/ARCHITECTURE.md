# ARCHITECTURE.md

## Назначение документа

Документ кратко описывает архитектуру REST-сервиса для работы с онлайн-подписками пользователей.

Цель — зафиксировать основные слои приложения, их ответственность и общий поток обработки запроса.

---

## Общий подход

Проект реализуется как простой монолитный REST-сервис.

Основной поток обработки запроса:

```text
Client
  ↓
HTTP Handler
  ↓
Service
  ↓
Repository
  ↓
PostgreSQL
```

---

## Слои приложения

### HTTP Handler

Отвечает за работу с HTTP:

- принимает запросы;
- читает path/query/body параметры;
- вызывает service;
- возвращает JSON-ответ;
- выставляет HTTP-коды ответов.

Handler не должен содержать бизнес-логику и SQL-запросы.

---

### Service

Отвечает за бизнес-логику:

- валидирует входные данные;
- применяет бизнес-правила;
- работает с датами;
- вызывает repository;
- формирует результат операции.

Service не должен знать детали HTTP и SQL.

---

### Repository

Отвечает за работу с базой данных:

- выполняет SQL-запросы;
- создаёт записи;
- читает записи;
- обновляет записи;
- удаляет записи;
- считает суммарную стоимость подписок.

Repository не должен содержать бизнес-логику HTTP-уровня.

---

## Основные модули

### Subscriptions

Модуль для управления подписками.

Отвечает за:

- создание подписки;
- получение подписки;
- получение списка подписок;
- обновление подписки;
- удаление подписки.

---

### Sum Cost

Модуль для подсчёта суммарной стоимости подписок.

Отвечает за:

- получение периода расчёта;
- фильтрацию по пользователю;
- фильтрацию по названию подписки;
- подсчёт итоговой суммы.

---

## База данных

Используется PostgreSQL.

Основная таблица:

```text
subscriptions
```

Основные поля:

- `id`;
- `service_name`;
- `price`;
- `user_id`;
- `start_date`;
- `end_date`;
- `created_at`;
- `updated_at`.

---

## Конфигурация

Конфигурация приложения хранится во внешнем файле:

```text
.env
```

или

```text
.yaml
```

В конфигурацию выносятся:

- порт приложения;
- параметры подключения к PostgreSQL;
- уровень логирования;
- режим запуска приложения.

---

## Логирование

Логи используются для отслеживания работы сервиса.

Логируются:

- запуск приложения;
- ошибки подключения к базе данных;
- создание, обновление и удаление подписок;
- ошибки при работе с подписками;
- параметры и результат подсчёта суммы подписок.

---

## Документация API

API описывается через Swagger.

Swagger должен содержать:

- список HTTP-ручек;
- request/response модели;
- query-параметры;
- возможные ошибки.

---

## Запуск проекта

Проект запускается через Docker Compose.

Минимальные сервисы:

- приложение;
- PostgreSQL.

---

## Что не входит в архитектуру

В рамках тестового задания не реализуются:

- авторизация;
- регистрация пользователей;
- роли;
- платежи;
- интеграции с внешними сервисами;
- микросервисная архитектура.
